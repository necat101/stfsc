# STFSC Engine: "Fuck Script" Ruleset & Technical Guide

This document is for future AI agents and developers working with the STFSC
engine scripting system.

This is where we get intimate with the engine, hence the name "Fuck Script" :3

## 1. System Overview

"Fuck Scripts" are dynamic entity behaviors built around the Rust trait
`FuckScript` in `src/world/scripting.rs`. Runtime scripts are native Rust
objects stored in the ECS as boxed trait objects.

The editor supports multiple script components on a single object. Keep the
preset scripts, but allow users to stack, edit, remove, disable, and add custom
scripts.

Hard rule: an entity may have at most `MAX_SCRIPTS_PER_ENTITY == 32` script
components. This constant lives in `src/project/scene.rs` and is enforced by
scene metadata, the editor, and `DynamicScript`.

## 2. Core Runtime Trait

Implement native scripts by implementing:

```rust
pub trait FuckScript: Send + Sync {
    fn on_start(&mut self, _ctx: &mut ScriptContext) {}
    fn on_update(&mut self, _ctx: &mut ScriptContext) {}
    fn on_fixed_update(&mut self, _ctx: &mut ScriptContext) {}
    fn on_late_update(&mut self, _ctx: &mut ScriptContext) {}
    fn on_enable(&mut self, _ctx: &mut ScriptContext) {}
    fn on_disable(&mut self, _ctx: &mut ScriptContext) {}
    fn on_collision_start(&mut self, _ctx: &mut ScriptContext, _other: Entity) {}
    fn on_collision_end(&mut self, _ctx: &mut ScriptContext, _other: Entity) {}
    fn on_trigger_start(&mut self, _ctx: &mut ScriptContext, _other: Entity) {}
    fn on_trigger_end(&mut self, _ctx: &mut ScriptContext, _other: Entity) {}
    fn on_player_respawn(&mut self, _ctx: &mut ScriptContext) {}
    fn on_ui_event(&mut self, _ctx: &mut ScriptContext, _event: &crate::ui::UiEvent) {}
    fn on_animation_event(&mut self, _ctx: &mut ScriptContext, _event_name: &str) {}
    fn on_xr_frame(&mut self, _ctx: &mut ScriptContext, _xr: &XrInputSnapshot) {}
    fn on_xr_input(&mut self, _ctx: &mut ScriptContext, _event: &XrInputEvent) {}
}
```

Use only the hooks you need. Empty defaults are intentional.

## 3. ScriptContext Helpers

Use `ScriptContext` instead of reaching around engine internals.

- `get_component<T>()`: read-only component access.
- `get_component_mut<T>()`: mutable component access.
- `transform()` / `transform_mut()`: quick access to this entity's `Transform`.
- `despawn_self()`: remove the current entity.
- `head_pose()`, `controller(hand)`, `xr_pose(space)`: tracked XR poses.
- `action_pressed(name)`, `action_just_pressed(name)`, `action_value(name)`: abstract XR actions.
- `request_haptic(hand, amplitude, duration_seconds)`: queue a haptic pulse.
- `track_transform_to_xr_pose(space, position_offset, rotation_offset)`: make an entity follow an XR pose.

## 4. Scene Metadata Model

`SceneEntity` has legacy script fields plus the new canonical component stack:

- `script: Option<String>`: legacy first script name.
- `scripts: Vec<String>`: legacy/runtime enabled script names.
- `script_components: Vec<ScriptComponent>`: canonical editable script slots.

Future agents should prefer `script_components` when editing scene files. Keep
the legacy fields in sync by calling `sync_legacy_script_fields()` after
changing components.

`ScriptComponent` fields:

- `name`: runtime script name. For presets this must match the registry name.
- `enabled`: disabled slots are preserved in the editor but omitted from runtime `SetScripts`.
- `source`: editable cache source shown in the inspector.
- `compile_mode`: `BuiltinNative`, `EditableNativeCache`, or `CustomNativeCache`.
- `cache_key`: optional future cache override.

Important helper behavior:

- `script_names()` returns enabled runtime names and clamps to 32.
- `set_script_names(names)` preserves existing source text by slot where possible.
- `ensure_script_components()` migrates legacy `script`/`scripts` into component slots.
- `clamp_script_components()` truncates at 32.

## 5. Script Stack Execution

Runtime `DynamicScript` contains `Vec<ScriptSlot>`, not just one script.

Scripts execute in stack order. Do not reorder scripts unless the user asks for
that. Order can matter for gameplay.

For each enabled slot, the engine:

1. Takes the slot out of `DynamicScript`.
2. Calls `on_start` and `on_enable` once if it has not started yet.
3. Delivers pending XR input events.
4. Calls `on_xr_frame`.
5. Delivers animation events.
6. Calls `on_fixed_update`.
7. Calls `on_update`.
8. Calls `on_late_update`.
9. Puts the slot back.

This is the required take pattern. Do not hold a mutable borrow of the
`DynamicScript` component while calling into a script, because scripts also need
mutable ECS access through `ScriptContext`.

## 6. Built-in Preset Scripts

Presets are first-class and should be kept available in the editor.

Current preset names:

- `TestBounce`: oscillating movement test.
- `CrowdAgent`: NPC wandering/steering with stuck recovery.
- `Vehicle`: physics vehicle bootstrap and suspension-style force logic.
- `PoliceAgent`: chase behavior around the player.
- `TrafficAI`: obstacle-aware vehicle AI.
- `WeaponNPC`: smooth facing/combat placeholder behavior.
- `CollisionLogger`: logs collision start/end events.
- `TouchToDestroy`: despawns itself on collision.
- `HeadAnchor`: follows the HMD/head pose.
- `LeftHandAnchor`: follows left grip pose.
- `RightHandAnchor`: follows right grip pose.
- `LeftAimAnchor`: follows left aim pose.
- `RightAimAnchor`: follows right aim pose.
- `TriggerHaptics`: trigger-edge haptic test.

If adding a new preset:

1. Implement the Rust struct in `src/world/scripting.rs`.
2. Implement `FuckScript`.
3. Register it in `GameWorld::new()` in `src/world/mod.rs`.
4. Add its name to `BUILTIN_SCRIPT_NAMES` in `src/world/scripting.rs`.
5. Add or update tests if behavior is non-trivial.

## 7. Custom FuckScript

The editor has a `Custom FuckScript` option. Its runtime name is
`CustomFuckScript`.

Current behavior:

- `ScriptComponent::custom()` creates an editable custom source template.
- Runtime deploy uses `CustomFuckScript`, a native placeholder script registered
  in the script registry.
- Export writes custom/edited source into the native script cache manifest.

Do not claim that arbitrary custom source is fully interpreted at runtime yet.
The current system preserves and exports custom source, and gives it a native
placeholder during live preview. The native codegen/binding layer is prepared
through export artifacts and can be extended later.

## 8. Editable FuckScript Source Syntax

This section is intentionally concrete. Cheaper/local agents should copy these
patterns instead of inventing new syntax.

Current status: this source is preserved in `ScriptComponent.source` and written
to the export script cache. Live preview still dispatches the registered native
runtime script name, usually `CustomFuckScript`, until generated native bindings
are connected. Treat these examples as the canonical editable/cache source style
for the native codegen path.

### 8.1 Minimal Script Shape

Always wrap behavior in one `script Name { ... }` block. The script name should
match the slot's runtime name when possible.

```fuckscript
script CustomFuckScript {
    on_start(ctx) {
        log("Custom script started");
    }

    on_update(ctx) {
        // Per-frame visual/gameplay logic.
    }

    on_fixed_update(ctx) {
        // Physics-step logic.
    }

    on_late_update(ctx) {
        // Camera, hand, and follow logic after normal update.
    }
}
```

Use `//` for comments. Use semicolons after assignments and function calls. Use
lower_snake_case for local variables. Keep field names explicit and boring.

### 8.2 Common Values And Helpers

Preferred literal/value style:

```fuckscript
let speed = 3.0;
let turn_speed = 8.0;
let up = vec3(0.0, 1.0, 0.0);
let identity = quat_identity();
```

Preferred context style:

```fuckscript
let dt = ctx.dt;
let self_pos = ctx.transform.position;
ctx.transform.position = self_pos + vec3(0.0, 1.0, 0.0) * dt;
```

Expected helper names for generated bindings:

- `ctx.transform.position`, `ctx.transform.rotation`, `ctx.transform.scale`
- `ctx.player_position()`
- `ctx.has_component("ComponentName")`
- `ctx.despawn_self()`
- `ctx.log(message)` or `log(message)`
- `ctx.action_pressed("name")`
- `ctx.action_just_pressed("name")`
- `ctx.action_value("name")`
- `ctx.head_pose()`
- `ctx.controller("left")`, `ctx.controller("right")`
- `ctx.xr_pose("head")`, `ctx.xr_pose("left_grip")`, `ctx.xr_pose("right_grip")`
- `ctx.request_haptic("left", amplitude, seconds)`
- `ctx.request_haptic("right", amplitude, seconds)`
- `ctx.track_transform_to_xr_pose(space, position_offset, rotation_offset)`

When a helper does not exist yet in generated bindings, map it to the nearest
native `ScriptContext` method instead of adding a second scripting API.

### 8.3 Move Up And Down

Simple visual movement should mutate `ctx.transform`. This is the safest starter
example.

```fuckscript
script HoverBob {
    on_start(ctx) {
        ctx.state.base_y = ctx.transform.position.y;
        ctx.state.time = 0.0;
    }

    on_update(ctx) {
        ctx.state.time = ctx.state.time + ctx.dt;

        let p = ctx.transform.position;
        let bob = sin(ctx.state.time * 2.0) * 0.25;
        ctx.transform.position = vec3(p.x, ctx.state.base_y + bob, p.z);
    }
}
```

### 8.4 Face The Player

This matches the native Rust example later in this file, but in editable source
form.

```fuckscript
script FacePlayer {
    on_update(ctx) {
        let player = ctx.player_position();
        let self_pos = ctx.transform.position;
        let to_player = player - self_pos;

        if length_squared(to_player) > 0.001 {
            let yaw = atan2(to_player.x, to_player.z);
            let target = quat_from_rotation_y(yaw);
            ctx.transform.rotation = slerp(
                ctx.transform.rotation,
                target,
                clamp(ctx.dt * 8.0, 0.0, 1.0)
            );
        }
    }
}
```

### 8.5 Patrol Between Two Points

Use persistent `ctx.state.*` fields for script-owned state. Do not create hidden
global variables.

```fuckscript
script PatrolLine {
    on_start(ctx) {
        ctx.state.origin = ctx.transform.position;
        ctx.state.target_index = 0;
    }

    on_update(ctx) {
        let a = ctx.state.origin + vec3(-3.0, 0.0, 0.0);
        let b = ctx.state.origin + vec3( 3.0, 0.0, 0.0);
        let target = if ctx.state.target_index == 1 { b } else { a };

        let pos = ctx.transform.position;
        let to_target = target - pos;

        if length(to_target) < 0.1 {
            ctx.state.target_index = 1 - ctx.state.target_index;
            return;
        }

        ctx.transform.position = pos + normalize(to_target) * 2.0 * ctx.dt;
    }
}
```

### 8.6 Physics Push

Physics behavior belongs in `on_fixed_update`. Prefer forces/velocities through
the physics helpers instead of teleporting dynamic bodies.

```fuckscript
script ConstantForwardForce {
    on_fixed_update(ctx) {
        if ctx.has_component("RigidBodyHandle") {
            let forward = ctx.transform.rotation * vec3(0.0, 0.0, -1.0);
            ctx.physics.add_force(ctx.entity, forward * 25.0);
        }
    }
}
```

If generated bindings do not expose `ctx.physics.add_force` yet, implement this
as a native Rust preset using `RigidBodyHandle` and `PhysicsWorld`.

### 8.7 Collision / Trigger Scripts

Collision hooks receive `other`, the other entity involved in the collision.

```fuckscript
script DestroyOnTouch {
    on_collision_start(ctx, other) {
        ctx.log("DestroyOnTouch hit something");
        ctx.despawn_self();
    }
}
```

```fuckscript
script OpenDoorTrigger {
    on_trigger_start(ctx, other) {
        ctx.state.open = true;
    }

    on_update(ctx) {
        if ctx.state.open == true {
            let target = quat_from_rotation_y(1.5708);
            ctx.transform.rotation = slerp(ctx.transform.rotation, target, ctx.dt * 4.0);
        }
    }
}
```

### 8.8 XR Hand, Head, And Haptics

Use `on_late_update` for tracked-pose following so the pose wins after normal
gameplay updates.

```fuckscript
script RightHandFollower {
    on_late_update(ctx) {
        ctx.track_transform_to_xr_pose(
            "right_grip",
            vec3(0.0, 0.0, 0.0),
            quat_identity()
        );
    }
}
```

Use `on_xr_input` or edge helpers for haptics.

```fuckscript
script TriggerPulse {
    on_update(ctx) {
        if ctx.action_just_pressed("trigger_right") {
            ctx.request_haptic("right", 0.7, 0.08);
        }

        if ctx.action_pressed("trigger_left") {
            ctx.request_haptic("left", 0.25, 0.02);
        }
    }
}
```

### 8.9 UI Event Script

Use exact button/action ids from the UI layout.

```fuckscript
script PauseMenuButtons {
    on_ui_event(ctx, event) {
        if event.kind == "button_clicked" && event.id == "resume" {
            ctx.menu_back();
        }

        if event.kind == "button_clicked" && event.id == "quit" {
            ctx.emit_engine_event("quit_requested");
        }
    }
}
```

If a UI helper is missing in native bindings, route through `on_ui_event` in a
native Rust preset and keep the source text as the exported cache version.

### 8.10 Source Syntax Rules For Small Agents

- Start with `script CustomFuckScript { ... }` unless the user names a script.
- Use only known hooks: `on_start`, `on_update`, `on_fixed_update`,
  `on_late_update`, `on_enable`, `on_disable`, `on_collision_start`,
  `on_collision_end`, `on_trigger_start`, `on_trigger_end`,
  `on_player_respawn`, `on_ui_event`, `on_animation_event`, `on_xr_frame`,
  `on_xr_input`.
- Use `ctx.state.name` for persistent script state.
- Use `ctx.transform` for simple movement/rotation/scale.
- Use `on_fixed_update` for physics and `on_late_update` for XR pose following.
- Do not invent new hook names like `Update`, `tick`, `awake`, or `process`.
- Do not write Python, Lua, JavaScript, or C# in `ScriptComponent.source`.
- If unsure, write a small script with one hook and one behavior.

## 9. Compile Cache / IL2CPP-Like Export Path

The engine is moving toward an IL2CPP-like model: resolve script stacks at build
time into native Rust-backed bindings for fast re-execution.

During export, `ProjectExporter::prepare_script_compile_cache()` writes:

- `script_cache/script_manifest.json`
- `script_cache/generated_script_bindings.rs`

The manifest includes scene path, entity id, slot index, script name, enabled
state, compile mode, source hash, cache key, and native symbol.

Future agents adding source-to-Rust generation should build on this manifest
instead of inventing a second cache format.

## 10. Deployment Rules

The editor sends runtime script stacks with:

```rust
SceneUpdate::SetScripts { id, names }
```

`names` must be enabled runtime names only, max 32. Unknown names are logged and
ignored by the runtime registry.

`SceneUpdate::AttachScript { id, name }` still exists, but for editor workflows
prefer `SetScripts` because it replaces the whole stack and preserves ordering.

## 11. Writing Good Runtime Scripts

- Scripts must be `Send + Sync`.
- Avoid allocations inside `on_update`; cache state inside the script struct.
- Prefer `on_start` for one-time component insertion/setup.
- Prefer `on_fixed_update` for physics-step logic and force application.
- Prefer `on_late_update` for camera/hand/pose followers.
- For simple movement, mutate `Transform`.
- For physics entities, apply forces through `RigidBodyHandle` and `PhysicsWorld`.
- Avoid nested ECS mutable borrows. Gather read data first, release borrows, then mutate.
- Keep user-facing behavior deterministic when possible.

## 12. Player Tracking

The runtime spawns a `Player` marker component and syncs its transform from the
desktop camera or XR head pose.

Find the player like this:

```rust
for (_entity, (transform, _player)) in ctx.world.query::<(&Transform, &Player)>().iter() {
    let player_pos = transform.position;
    // Use player_pos here.
}
```

## 13. Collision And Trigger Events

Collision callbacks are dispatched through Rapier events. Rigid bodies must have
their `user_data` set to `entity.to_bits().get() as u128` for collision events
to map back to ECS entities.

Use:

- `on_collision_start`
- `on_collision_end`
- `on_trigger_start`
- `on_trigger_end`

## 14. Example Native Script

```rust
pub struct FacePlayerScript {
    turn_speed: f32,
}

impl Default for FacePlayerScript {
    fn default() -> Self {
        Self { turn_speed: 8.0 }
    }
}

impl FuckScript for FacePlayerScript {
    fn on_update(&mut self, ctx: &mut ScriptContext) {
        let mut player_pos = None;
        for (_entity, (transform, _player)) in ctx.world.query::<(&Transform, &Player)>().iter() {
            player_pos = Some(transform.position);
            break;
        }

        let Some(player_pos) = player_pos else {
            return;
        };

        if let Some(mut transform) = ctx.transform_mut() {
            let to_player = player_pos - transform.position;
            if to_player.length_squared() > 0.001 {
                let target_yaw = to_player.x.atan2(to_player.z);
                let target = glam::Quat::from_rotation_y(target_yaw);
                transform.rotation = transform
                    .rotation
                    .slerp(target, (ctx.dt * self.turn_speed).clamp(0.0, 1.0));
            }
        }
    }
}
```

Register it:

```rust
reg.register("FacePlayer", || FacePlayerScript::default());
```

Then add `"FacePlayer"` to `BUILTIN_SCRIPT_NAMES` if it should appear as an
editor preset.

## 15. Agent Checklist

When an AI agent writes or edits scripts:

1. Decide whether this is a built-in native preset or a custom editable script.
2. Keep script stacks under 32 components.
3. Preserve existing preset scripts unless the user explicitly asks to remove them.
4. Keep `script_components`, `script`, and `scripts` synchronized.
5. Register new native presets in both the registry and `BUILTIN_SCRIPT_NAMES`.
6. Update export/cache behavior only through the existing `script_cache` manifest path.
7. Run at least `cargo check --bins` and preferably `cargo test`.

---

Always remember: runtime Rust scripts are native and fast; editable/custom source
is currently preserved and exported for native cache generation, with live
preview backed by the registered runtime script name.
